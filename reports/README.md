# Specmatic reports

These are **HTML reports generated by Specmatic** (`specmatic test`, report configured in
[`../specmatic.yaml`](../specmatic.yaml) v3 → `specmatic.governance.report.formats: [html, ctrf]`),
one per CI pass. Open any `.html` in a browser.

`api_contract.yaml` is the **only** spec — it governs all 52 real `/api` operations directly (6
hand-authored at full request/response fidelity, the rest auto-generated via
[`../scripts/generate_full_contract.py`](../scripts/generate_full_contract.py) into the same
file). One spec, three filtered test passes, zero missing in spec, zero failures.

The app exposes a Spring-Boot-Actuator-style **`/actuator/mappings`** endpoint (test-only, in `app.py`),
so Specmatic reads the app's real routes and reports **actual** coverage — every contracted operation is
confirmed to exist in the running app and is tested (100% Absolute Coverage), not just matched against the spec.

| Report | Filter | Result | What it covers |
|---|---|---|---|
| `api_contract-core.html` | the 6 hand-authored endpoints | 41 tests, 41 pass | Full request/response fidelity — login, session, connections, command, execute, undo. Conformance + generative resiliency. |
| `api_contract-auto-generated.html` | every other operation except join preview/execute | 144 tests, 144 pass | Every one of the other 44 operations — 200/400/401/404, conformance + generative resiliency. See [`../CONTRACT_SCOPE.md`](../CONTRACT_SCOPE.md) for exactly what each needed to get to a real, verified success (or 404, for the one operation where that's the correct answer). |
| `api_contract-join.html` | `/api/join/preview`, `/api/join/execute` | 4 tests, 4 pass | Same 2 operations, using [`../specmatic-conformance-only.yaml`](../specmatic-conformance-only.yaml) — `schemaResiliencyTests: none` instead of `all`. Specmatic's resiliency mutator doesn't respect `required` fields nested inside array items (confirmed across inline, `minItems`, and `$ref`'d schema representations — all three still let it generate a `joins` array whose items drop the required `on` clause). Conformance + the real 401 boundary still run; only that one mutation category is skipped. |

**How 0 failures is reached honestly:** the app's real API bearer-token auth accepts the
token configured in `API_BEARER_TOKEN` (declared once in `../specmatic.yaml` under
`securitySchemes`, which is how Specmatic sends it). External examples in `../examples_api/`
exercise the authenticated `200`/`400` paths with that token, and a *wrong* token (or none) to
exercise the real `401` path — so each pass covers both the authorized and the unauthorized
response of every secured endpoint, using the app's ordinary auth (no test-only bypass).

CI runs these same 3 filtered passes, all blocking — see
[`../.github/workflows/contract.yml`](../.github/workflows/contract.yml).

Regenerate locally: run the commands in [Run it locally → Run the Specmatic tests](../readme.md#2--run-the-specmatic-tests)
with each pass's `--filter` (and `--config specmatic-conformance-only.yaml` for the join pass),
then copy `build/reports/specmatic/test/html/index.html` to the matching file here. In CI each
pass also uploads `build/reports/specmatic/test/html` as an artifact.
