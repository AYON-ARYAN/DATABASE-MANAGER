# Specmatic reports

These are **HTML reports generated by Specmatic** (`specmatic test`, report configured in
[`../specmatic.yaml`](../specmatic.yaml) v3 → `specmatic.governance.report.formats: [html, ctrf]`),
one per CI test job. Open any `.html` in a browser. **All four jobs are at 100% API coverage.**

The app exposes a Spring-Boot-Actuator-style **`/actuator/mappings`** endpoint (test-only, in `app.py`),
so Specmatic reads the app's real routes and reports **actual** coverage — every contracted operation is
confirmed to exist in the running app and is tested (100% Absolute Coverage), not just matched against the spec.

| Report | Spec | Mode | Coverage | What it covers |
|---|---|---|---|---|
| `contract_public-contract.html`   | contract_public.yaml | contract   | 100% | value-checked conformance (external examples: 3 roles + rejection + 400) |
| `contract_public-resiliency.html` | contract_public.yaml | resiliency | 100% | generative negative/boundary tests |
| `api_contract-contract.html`      | api_contract.yaml    | contract   | 100% | every response of every endpoint — 200 / 400 / 401 — LLM mocked |
| `api_contract-resiliency.html`    | api_contract.yaml    | resiliency | 100% | generative negatives across the full API |

**How 100% on the protected API is reached honestly:** the test-auth gate accepts one exact
bearer token (the value of `SPECMATIC_TEST`). External examples in `../examples_api/` carry that
token to exercise the authenticated `200`/`400` paths, and carry a *wrong* token (or none) to
exercise the real `401` path — so a single authenticated run covers both the authorized and the
unauthorized response of every secured endpoint.

Regenerate locally: start the app + LLM stub, then
`bash scripts/gen_specmatic_reports.sh` (see `scripts/local_specmatic_boot.sh`).
In CI each job also uploads its `build/reports/specmatic/test/html` as an artifact.
